Design: Per-Task Model Routing

Status: Implemented · Version: 1.0 · Date: 2026-05-13

Abstract: Assign each LLM-consuming task in the stock analysis pipeline its own model, selected by task characteristics — reasoning depth, throughput, cost sensitivity, and output format requirements. Replaces the previous single-model-for-all approach (LITELLM_MODEL only) with a layered override system.

1. Motivation

1.1 Problem Statement

Prior to this design, all LLM calls in the system shared one global model (LITELLM_MODEL). The Agent had an optional independent model (AGENT_LITELLM_MODEL), but all other tasks — deep stock analysis, batch garbage-stock screening, theme detection, market review — used the same model.

This created a tension:

  • Using a powerful (expensive) model for everything wastes money on simple tasks (text classification, markdown generation) that a local or cheaper model handles fine.
  • Using a cheap (local) model for everything degrades the core stock analysis quality where reasoning depth directly impacts prediction accuracy and JSON parsing reliability.

1.2 Design Goals

  1. Per-task model assignment: Each task type independently configurable via environment variable.
  2. Backward compatible: All new config fields default to empty; when empty, the task inherits LITELLM_MODEL.
  3. Minimal code intrusion: Model override flows through existing method signatures.
  4. Observable degradation: When a weaker model produces invalid output, the parser logs the model identity alongside the failure.

Config dataclass

  config.gold_digger_model      = "ollama/qwen3.5:9b"
  config.scanner_model          = "deepseek/deepseek-v4-flash"

  config.market_review_model    = "deepseek/deepseek-v4-flash"
       │
       ├──→ analyzer.analyze(model=config.litellm_model)           [Stock Analysis]
       ├──→ gold_digger: generate_text(model=gold_digger_model)    [Gold Digger]
       ├──→ market_scanner: process_single_stock(model_override=)  [Scanner]
       ├──→ theme_detector: generate_text(model=theme_model)       [Theme]
       └──→ market_review: generate_text(model=review_model)       [Review]
       │
       └──→ GeminiAnalyzer._call_litellm(model=override)
              └──→ _dispatch_litellm_completion(model, ...)
                     └──→ LiteLLM Router (from litellm_config.yaml)
                            ├──→ DeepSeek V4 Pro
                            ├──→ DeepSeek V4 Flash
                            └──→ Ollama qwen3.5:9b
Key invariant: All five models exist in litellm_config.yaml's model_list. Therefore _dispatch_litellm_completion always routes through the LiteLLM Router, which handles API key resolution, base URL, and retries per the YAML config.

2.3 Override Resolution

In GeminiAnalyzer._call_litellm():

primary_model = model or config.litellm_model
models_to_try = [primary_model] + (config.litellm_fallback_models or [])

When model is None, behavior is identical to previous code. When a per-task model is provided, it replaces the primary slot while preserving the global fallback chain.

3.2 Parser Resilience

Risk: Weaker models (Ollama) may produce malformed JSON more frequently. The system must not crash.

Both JSON-dependent parsers include model-tagged fallback:

  • _parse_llm_gold_response(): On JSONDecodeError, returns neutral defaults (confidence=40) + model-tagged WARNING log.
  • _parse_themes(): On JSONDecodeError, returns hardcoded fallback themes + model-tagged WARNING log.

6. Edge Cases & Rollback

Model Not in YAML Config

If a per-task model is not registered in litellm_config.yaml, the Router bypass fails and the call falls through to direct litellm.completion(). Always register per-task models in the YAML config.

Ollama JSON Reliability

Monitor logs for JSON parse failed warnings. If failure rate exceeds 20%, switch the affected task to a cloud model.

Fallback Chain

Per-task override uses the global LITELLM_FALLBACK_MODELS chain. No per-task fallback list.

Rollback

Set all per-task variables to empty in .env. All tasks automatically inherit LITELLM_MODEL. No code changes required.
