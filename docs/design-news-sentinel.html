1. 需求概述与目标

1.1 背景

当前系统的新闻获取策略是被动拉取（pull-on-demand）：每次分析单只股票时，临时调用第三方搜索 API（Tavily / SerpAPI / Bocha / EastMoney RSS 等）获取最新资讯。这种模式的缺陷：

  • 每次分析耗时长，受 API 限速和网络延迟影响
  • 无法主动感知市场突发事件，缺乏预判能力
  • 无法跨股票/行业做宏观归纳，找到共同催化剂
  • 对第三方 API 依赖高，成本不可控

1.2 目标

News Sentinel 是一个独立的主动情报服务，目标是：

  1. 周期性地直接抓取行业、财经、政策、监管、突发等各类网站，无需 API Key
  2. 使用 LLM 对每条新闻进行分类标注和市场影响优先级评分
  3. 将新闻本地缓存，向股票分析服务提供高效查询接口
  4. 周期性淘汰低价值过期内容，长期索引高影响力新闻作为知识底座
  5. 每轮抓取完成后执行综合宏观分析，若发现市场机会则主动通知股票分析服务触发深度分析

1.3 设计原则

  • 渐进增强：服务不可用时，现有股票分析流程自动降级至原有 SearchService，不阻断主流程
  • 成本优先：优先免费渠道（RSS / 直接抓取），LLM 调用批量化、有限速
  • 最小侵入：以独立进程运行，通过 REST 接口与主服务通信，不共享进程内存
  • 可审计：每条新闻保留来源、抓取时间、LLM 分析理由，便于回溯

2.2 部署模式

模式 A：内嵌进程（初期推荐）

作为 main.py --sentinel 子进程或后台线程运行，随主服务启动。通信通过 SQLite 共享数据库 + 内部函数调用。

  • 部署简单，无需额外端口
  • 适合单机 / Docker 单容器场景

模式 B：独立微服务（长期目标）

独立进程运行，通过 HTTP REST API 通信。可独立扩展、重启，支持多实例。

  • 需要 uvicorn server_sentinel:app
  • 适合生产环境 / Docker Compose 多容器

3. 核心组件设计

3.1 Spider Engine（抓取引擎）

基于 httpx + BeautifulSoup4（轻量，无需 Scrapy 进程模型）实现可插拔的 Spider 注册表。每个 Spider 对应一个数据源，实现统一的 SpiderBase 接口。

为何不用 Scrapy 框架
Scrapy 是进程级框架，与 asyncio 集成复杂，且现有代码库基于 threading。初期使用 httpx.AsyncClient + asyncio 更容易集成进现有调度器，成熟后再迁移。 
# src/services/sentinel/spiders/base.py
class SpiderBase(ABC):
    name: str                     # e.g. "eastmoney_rss"
    category: str                 # finance / gov / industry / breaking
    interval_minutes: int = 30    # per-spider scrape interval
    max_items_per_run: int = 50
    timeout_seconds: int = 10

    @abstractmethod
    async def fetch(self) -> list[RawArticle]:
        """Fetch raw articles from source. Returns deduplicated by URL."""
        ...

    def is_enabled(self, config: SentinelConfig) -> bool:
        return True

# RawArticle (unprocessed)
@dataclass
class RawArticle:
    url: str
    title: str
    content: str        # full text or excerpt
    published_at: datetime | None
    source_name: str    # "东方财富"
    source_url: str     # base domain
    spider_name: str
    fetched_at: datetime = field(default_factory=datetime.utcnow)
    language: str = "zh"   # zh / en

3.2 Processing Pipeline（处理流水线）

RawArticle → Deduplicator → ContentNormalizer → LLMClassifier → NewsItem

3.2.1 Deduplicator

  • SHA-256 of normalize(url) as primary key
  • SimHash of title + content[:200] for near-duplicate detection（阈值 Hamming distance ≤ 3）
  • Already-seen items are skipped before LLM call

3.2.2 ContentNormalizer

  • 清理 HTML 标签、广告锚文本、导航菜单文本
  • 截断内容至 2000 字符（LLM 上下文预算）
  • 语言检测（langdetect），标注 language
  • 时间规范化为 UTC datetime

3.2.3 LLM Classifier（批量分类）

每批 10-20 条新闻聚合成单次 LLM 调用，降低 API 成本。

# 单条新闻 LLM 分类 prompt 示例（批量版嵌入 JSON array）
"""
你是专业的金融新闻分析师。请对以下新闻进行分类和评估，返回 JSON 数组。

每条新闻输出字段：
- category: finance|policy|industry|regulatory|macro|breaking|company|other
- priority: 1(低影响)~5(极高影响，可能引发市场波动)
- sentiment: bullish|bearish|neutral
- market_scope: stock_specific|sector|market_wide|macro
- affected_sectors: ["半导体","新能源",...] 最多3个
- affected_stocks: ["600519","NVDA",...] 仅明确提及的代码
- impact_horizon: intraday|short_term(1-5d)|medium_term(1-4w)|long_term(1m+)
- reasoning: 20字内说明为何给此优先级
- is_actionable: true/false  是否需要立即触发股票分析

新闻列表：
{news_json_array}
"""
LLM 调用成本控制
  • 每轮最多分类 200 条（超出排队到下一轮）
  • 低优先级来源（如博客聚合）只在非高峰时段分类
  • 已分类内容 24h 内不重新分类（除非内容发生变化）
  • 使用系统中已配置的 LLM（本地模型优先，节省 API 费用）

3.3 News Store（新闻存储）

使用 SQLite（与主 DB 分离）作为持久层，可选 Redis 作为热缓存。

# 数据库路径（独立，不与主 DB 合并）
SENTINEL_DB = data/sentinel.db

3.4 TTL Purger（过期清理）

优先级保留时长清理策略
P5（极高影响）永久保留只标记 is_archived=true，不删除
P4（高影响）90 天超期后降为 archived
P3（中等）30 天超期后软删除
P2（低）7 天硬删除
P1（极低）24 小时硬删除

3.5 Comprehensive Analyzer（综合分析器）

每轮抓取完毕后运行一次，聚合最近 N 条高优先级新闻（P3+），由 LLM 做宏观归纳。 

# 综合分析 prompt 骨架
"""
以下是过去 {window} 小时内的重要市场资讯（共 {count} 条，已按优先级排序）：
{news_digest}

请完成：
1. 宏观主题归纳（最多3条）：当前市场最主要的催化剂或风险
2. 板块机会信号：哪些板块可能在未来1-5个交易日出现异动
3. 具体股票线索（如有）：值得立即分析的股票代码及理由
4. 风险提示：需警惕的重大系统性风险
5. 综合判断：整体市场情绪（极度悲观/悲观/中性/乐观/极度乐观）

输出 JSON，字段：themes, sector_opportunities, stock_leads, risk_alerts, market_mood
"""

若 stock_leads 非空且置信度 ≥ 0.7，自动向主服务发送分析触发通知。

4. 数据模型

4.1 news_items（新闻条目）

CREATE TABLE news_items (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    url_hash        TEXT NOT NULL UNIQUE,          -- SHA-256(normalized_url)
    simhash         INTEGER,                        -- 64-bit SimHash for near-dedup
    url             TEXT NOT NULL,
    title           TEXT NOT NULL,
    content         TEXT,                           -- first 2000 chars
    source_name     TEXT NOT NULL,
    source_url      TEXT NOT NULL,
    spider_name     TEXT NOT NULL,
    language        TEXT DEFAULT 'zh',

    -- LLM classification
    category        TEXT,      -- finance/policy/industry/regulatory/macro/breaking/company/other
    priority        INTEGER,   -- 1-5
    sentiment       TEXT,      -- bullish/bearish/neutral
    market_scope    TEXT,      -- stock_specific/sector/market_wide/macro
    affected_sectors TEXT,     -- JSON array
    affected_stocks  TEXT,     -- JSON array of stock codes
    impact_horizon   TEXT,     -- intraday/short_term/medium_term/long_term
    llm_reasoning   TEXT,      -- 20-char reasoning
    is_actionable   INTEGER DEFAULT 0,

    -- lifecycle
    published_at    TEXT,      -- ISO8601 UTC
    fetched_at      TEXT NOT NULL,
    analyzed_at     TEXT,
    expires_at      TEXT,      -- computed from priority + published_at
    is_expired      INTEGER DEFAULT 0,
    is_archived     INTEGER DEFAULT 0,

    CONSTRAINT priority_range CHECK (priority BETWEEN 1 AND 5)
);

CREATE INDEX idx_news_priority    ON news_items(priority DESC, fetched_at DESC);
CREATE INDEX idx_news_category    ON news_items(category, is_expired);
CREATE INDEX idx_news_stocks      ON news_items(affected_stocks);  -- JSON, use LIKE for now
CREATE INDEX idx_news_fetched     ON news_items(fetched_at DESC);
CREATE VIRTUAL TABLE news_fts USING fts5(
    title, content, source_name,
    content='news_items', content_rowid='id'
);

6. LLM 分析流水线

6.1 分类批次策略

                 每批 ≤ 20 条
                 每次 LLM 调用
                    ┌───────┐
Raw Articles ──────▶│ Batch │──▶ LLM Call ──▶ Classification Result
   (N 条)     split └───────┘               (JSON array N items)
              into
              ceil(N/20) batches
  • 批次间隔：1 秒（避免 API 限速）
  • 重试策略：3 次，指数退避
  • 若 LLM 调用失败：标记 analyzed_at=null，下次循环重试
  • 使用与主服务相同的 LLM 配置（复用 GeminiAnalyzer 底层调用）

6.2 优先级评分公式

LLM 返回原始 priority（1-5），结合规则调整：

final_priority = clip(
    llm_priority
    + gov_source_bonus(+1 if category='policy' and source in GOV_SOURCES)
    + freshness_bonus(+1 if age < 1h)
    + recurrence_penalty(-1 if similar item seen in last 6h)
    , 1, 5
)

6.3 综合分析触发条件

触发条件说明
本轮新增 P4/P5 新闻 ≥ 3 条高影响新闻集中涌现
单一板块 P3+ 新闻 ≥ 5 条板块集中事件
任意 P5 新闻出现极端事件（如央行降息、重大政策）
定时触发（每 4 小时）常规宏观梳理，即使无高影响新闻

7.2 抓取轮次（Cycle）生命周期

START ──▶ spawn spiders (parallel, max_concurrency=5) │ ├── spider_1: fetch → normalize → deduplicate → queue ├── spider_2: fetch → normalize → deduplicate → queue └── ... │ ▼ all spiders done OR timeout (CYCLE_TIMEOUT_MINUTES=10) │ ▼ batch_classify(queued_articles, batch_size=20) │ ▼ update news_store (insert / update priority) │ ▼ evaluate_cycle_trigger_conditions() │ ┌────┴────┐ yes no │ │ ▼ ▼ run_comprehensive_analysis() skip, log │ ▼ check stock_leads.confidence │ ≥ 0.7 │ ▼ notify_stock_service(codes) │ ▼ END CYCLE

7.3 优雅停机

  • 收到 SIGTERM/SIGINT 时，完成当前抓取批次，跳过 LLM 分类，安全写库后退出
  • 未分类文章保留 analyzed_at=null，下次启动续处理

NotificationServiceSentinel 借用现有渠道发送 policy/breaking 通知零（直接调用现有接口）
DatabaseManagerSentinel 使用独立 DB，不共享零
Web UIPhase 4 新增"情报中心"侧边栏页面中（新路由 + 新组件）

10.2 降级策略

零停机降级
Sentinel 未部署或不可用时，所有调用路径自动回落到原有 SearchService 在线 API 调用。主服务中 SENTINEL_ENABLED=false（默认）即可完全禁用，无任何副作用。

13. 风险与缓解

风险概率影响缓解措施
目标网站反爬，抓取被封锁 中 中 设置合理 User-Agent 和 delay；优先 RSS（无反爬）；失败后标记 Spider 为 disabled，人工介入
LLM 分类质量不稳定（本地模型） 高 中 保留原始 LLM 输出；规则后处理（e.g. 政府来源自动 +1 priority）；人工标注 ground truth 用于回归
综合分析误报（虚假机会信号） 中 高 置信度阈值 ≥ 0.70；要求 ≥ 2 条独立来源支持；用户可配置 SENTINEL_TRIGGER_CONFIDENCE；触发通知前记录推理链
DB 膨胀（新闻量大） 中 低 TTL Purger 每日清理；低优先级内容 7 天硬删；content 截断 2000 字；可配置 SENTINEL_DB_MAX_SIZE_MB 强制清理
LLM API 费用超出预期 中 中 SENTINEL_LLM_MAX_PER_CYCLE 限制批次；优先使用本地模型（Ollama）；低频时段降级为规则分类
与主服务进程竞争 CPU/内存 低 中 独立进程部署（Phase 5）；并发 Spider 数量可配置；LLM 调用与主服务异步解耦
法律合规（版权/robots.txt） 低 高 遵守 robots.txt；内容仅内部使用；不爬取付费墙内容；保留来源 URL 用于归因
下一步行动
  1. 确认技术选型：httpx + BeautifulSoup4（vs Scrapy）—— 如需 JS 渲染场景支持，后续可引入 playwright
  2. 确认部署模式：Phase 1 内嵌进程，Phase 5 拆分微服务
  3. 确认优先接入的抓取源（至少 5 个 RSS 源作为 Phase 1 基线）
  4. 评估是否需要 Redis 热缓存（单机 SQLite FTS5 通常足够）
  5. 明确 Web UI Phase 4 的 UI 设计（情报中心侧边栏 vs 独立页面）
