News Sentinel — 部署与运维指南

版本：Phase 5  |  更新：2026-05-15  |  文件：docs/design-news-sentinel-deployment.html

本文档说明 News Sentinel 的各种部署模式、环境变量配置、Spider 扩展方法、运维监控接入和常见故障排查。架构设计请参阅 docs/design-news-sentinel.html。

1. 部署模式总览

模式启动方式适用场景
嵌入模式 SentinelScheduler 在主进程中启动后台线程 本地开发、资源受限环境；与主分析进程共享同一 Python 进程
独立进程 python server_sentinel.py 生产环境首选；内置 HTTP 健康端点，可被 Docker/k8s 探活
Docker Compose sentinel service in docker-compose.yml 与主服务共同编排；挂载同一 SQLite 数据库目录
单次运行 python -m src.services.sentinel.service 定时任务（cron）、调试、CI 冒烟测试
Dry-run python server_sentinel.py --dry-run 验证 Spider 连通性，不写入数据库

2. 快速启动

2.1 本地开发（嵌入模式）

# 1. 在 .env 中启用 Sentinel
SENTINEL_ENABLED=true
SENTINEL_DB_PATH=data/sentinel.db

# 2. 正常启动主服务，Sentinel 在后台随 main.py 运行
python main.py --serve

2.2 独立进程

# 前台运行，默认监听 :9100
python server_sentinel.py

# 自定义端口
python server_sentinel.py --port=9200

# 只跑一次然后退出（不启动 HTTP 服务器）
python -m src.services.sentinel.service

# 仅抓取，不写库
python -m src.services.sentinel.service --dry-run

# 持续循环模式（无 HTTP 服务器）
python -m src.services.sentinel.service --loop

2.3 Docker Compose 示例

services:
  sentinel:
    build: .
    command: python server_sentinel.py
    environment:
      SENTINEL_ENABLED: "true"
      SENTINEL_DB_PATH: /data/sentinel.db
      SENTINEL_RSSHUB_BASE_URL: http://rsshub:1200
      SENTINEL_SERVER_PORT: "9100"
    volumes:
      - sentinel_data:/data
    ports:
      - "9100:9100"
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:9100/healthz"]
      interval: 60s
      timeout: 5s
      retries: 3

  rsshub:
    image: diygod/rsshub
    ports:
      - "1200:1200"
    restart: unless-stopped

volumes:
  sentinel_data:
提示：若主服务（server.py）和 server_sentinel.py 需共享 SQLite 数据库，请确保两者挂载同一目录，并将 SENTINEL_DB_PATH 设置为一致路径。SQLite WAL 模式下两个进程可以并发读写。

3. 环境变量参考

变量默认值说明
SENTINEL_ENABLEDfalse主开关；false 时所有 Sentinel 组件静默，不影响主流程
SENTINEL_DB_PATHdata/sentinel.dbSQLite 数据库路径，支持绝对路径
SENTINEL_SERVER_PORT9100server_sentinel.py 内置 HTTP 服务监听端口
SENTINEL_RSSHUB_BASE_URLhttp://localhost:1200本地 RSSHub 实例地址；空则所有 RSSHubSpider 跳过
SENTINEL_RSSHUB_TIMEOUT15RSSHub 请求超时（秒）
SENTINEL_CYCLE_INTERVAL_MINUTES30完整抓取轮次间隔（分钟）
SENTINEL_ANALYSIS_INTERVAL_HOURS4综合宏观分析触发间隔（小时）
SENTINEL_LLM_BATCH_SIZE20每批 LLM 分类新闻数量
SENTINEL_LLM_MAX_PER_CYCLE200每轮最多分类条数，防止 LLM 费用失控
SENTINEL_TRIGGER_CONFIDENCE0.70自动触发股票深度分析的置信度阈值（0.0–1.0）
SENTINEL_CYCLE_TIMEOUT_MINUTES10单轮抓取最大超时（分钟）
SENTINEL_MAX_SPIDER_CONCURRENCY5并发 Spider 数量（当前为顺序执行，预留参数）
SENTINEL_REQUEST_DELAY_SECONDS2.0同一站点请求间隔（秒），礼貌爬取
SENTINEL_WEBHOOK_SECRET（空）微服务模式下 webhook 鉴权密钥（预留）
SENTINEL_REDIS_URL（空）可选 Redis 热缓存 URL（预留，空则不使用）
SENTINEL_ENABLED_SPIDERSall逗号分隔的 Spider name 白名单，或 all
SENTINEL_TRADING_HOURS_BOOSTtrue交易时段内提升抓取频率（预留）

4. HTTP 健康与指标端点（server_sentinel.py）

路径响应用途
GET /healthz {"status": "ok", "uptime_seconds": N} Docker/k8s liveness/readiness 探针
GET /metrics 完整 SentinelMetrics.summary() JSON 监控系统抓取（Prometheus textfile export、Grafana JSON）
GET /status 配置摘要 + DB 统计 + Spider 列表 运维快速诊断

Prometheus textfile 示例

9. 常见故障排查

Spider 持续返回空结果（健康度降级）

每个 Spider 连续 3 次空抓取后标记为 DEGRADED，在 --dry-run 摘要或 /status 端点可见。常见原因：

  • RSS 源网络不可达 — 检查防火墙/代理
  • RSSHub 实例未启动 — curl http://localhost:1200/healthz
  • 政府网站 HTML 结构变化 — 需更新对应 Spider 的 _parse_items()
LLM 分类停止工作
  • 检查主 LLM 配置（GEMINI_API_KEY 或 LiteLLM 路由）是否有效
  • 查看日志中 LLM classification step failed 的详细 traceback
  • SENTINEL_LLM_MAX_PER_CYCLE=0 可临时禁用分类，不影响抓取
DB 文件持续增长
  • TTL 清理每轮自动运行，检查 /metrics 的 ttl_purge.total_deleted 是否在增加
  • 大量 P5 新闻永久保留 — 可调低 LLM 分配 P5 的频率（通过 Prompt 调整）
  • 运行 sqlite3 data/sentinel.db "VACUUM;" 回收碎片
综合分析不触发
  • 确认 SENTINEL_ANALYSIS_INTERVAL_HOURS 设置合理（默认 4 小时）
  • 检查数据库中是否有足够的 P3+ 已分类新闻（SELECT COUNT(*) FROM news_items WHERE priority >= 3 AND sentiment IS NOT NULL;）
  • 查看日志中 Comprehensive analysis step failed
server_sentinel.py 端口冲突

设置 SENTINEL_SERVER_PORT=9200（或其他空闲端口），或用 --port=9200 命令行参数覆盖。

10. 回滚方式

注意：Sentinel 默认关闭（SENTINEL_ENABLED=false）。若出现问题，最简单的回滚是将 SENTINEL_ENABLED 改回 false 并重启主服务，Sentinel 相关代码路径完全静默。
  • 主服务回滚：将 SENTINEL_ENABLED=false 写入 .env，重启 server.py。SearchService 会跳过 Sentinel 缓存，直接走在线搜索。
  • 独立进程回滚：停止 server_sentinel.py 进程，主服务不受影响。
  • 数据库回滚：用备份文件替换 data/sentinel.db，重启 Sentinel 服务。
  • 代码回滚：git revert 对应 Sentinel commits，Sentinel 模块对主流程无硬依赖，回滚不影响分析主链路。
