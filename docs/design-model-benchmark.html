1. Motivation & Problem Statement

The system supports multiple LLM backends (Anthropic Claude, Google Gemini, OpenAI GPT, DeepSeek, Ollama local models, and OpenAI-compatible proxies like Anspire/AIHubMix). Each model produces different stock analysis results — different operation_advice, decision_type, stop_loss, and take_profit levels.

Core question: Which model produces the most actionable and accurate predictions for US stocks?

Manual comparison is infeasible because:

  • Each analysis run costs money (API fees) and time (LLM latency)
  • Subjective judgment ("this report reads well") does not correlate with actual prediction accuracy
  • US market dynamics differ from CN — a model strong on A-shares may underperform on US equities

Solution: An automated benchmarking system that runs identical analyses through multiple models, backtests each prediction against actual future price data, and ranks models by a composite accuracy score.

3. Scoring Formula

The composite score is a weighted sum of five normalized metrics, scaled to 0–100:

composite_score = 100 x ( direction_accuracy_norm * 0.35 + win_rate_norm * 0.30 + excess_return_norm * 0.20 + consistency_norm * 0.10 + conviction_bonus * 0.05 )

Metric Definitions

MetricWeightRaw ValueNormalization
Direction Accuracy 35% % of predictions where direction_correct=true Linear: raw% ÷ 100
Win Rate 30% win ÷ (win + loss), neutrals excluded Linear: raw% ÷ 100
Excess Return 20% Mean actual stock_return_pct across all completed evals Sigmoid: 1/(1+e-r/5) — maps +10%→0.88, 0%→0.50, -10%→0.12
Consistency 10% 1 − CV where CV = |stdev(returns) / mean(returns)| Clamp: max(0, 1 − CV/2). CV=0→1.0, CV≥2→0.0
Conviction Bonus 5% Rewards decisive models that are correct; penalizes bold-but-wrong decisive_ratio*0.4 + correct_when_decisive*0.6

Conviction Bonus Detail

A model is "decisive" when it outputs position_recommendation=long (i.e., it actually recommends buying, not just holding cash):

decisive_ratio = count(position=long) / total_analyses
correct_ratio  = count(decisive AND win) / count(decisive AND (win OR loss))
conviction     = decisive_ratio * 0.4 + correct_ratio * 0.6

A model that always says "hold" gets conviction = 0.5 (neutral). A model that takes bold long positions and is right gets > 0.5. A model that takes bold long positions and is wrong gets < 0.5.

4. Three-Phase Workflow

Phase 1: Analyze (Day 0)

# Sequential (default) — one model at a time, one stock at a time
python -m src.services.model_benchmark --analyze --stocks AAPL,NVDA,MSFT

# Parallel — models sequential, stocks concurrent (faster)
python -m src.services.model_benchmark --analyze --stocks AAPL,NVDA,MSFT --parallel

# Parallel + Debug — live side-by-side comparison as results come in
python -m src.services.model_benchmark --analyze --stocks AAPL,NVDA --parallel --debug

# Limit concurrency
python -m src.services.model_benchmark --analyze --stocks AAPL,NVDA --parallel --max-parallel 2
  1. discover_models() reads config.llm_model_list (aggregated from LLM_CHANNELS + legacy env vars)
  2. Work items are grouped by model_id; models are processed sequentially (for env-var safety via _ModelOverride)
  3. Within each model group, stocks are dispatched concurrently via ThreadPoolExecutor (if --parallel is set)
  4. build_agent_executor() creates a fresh agent with the overridden model
  5. Full pipeline runs (Technical → Intel → Decision) for each stock
  6. If --debug: latency (time.perf_counter()), token usage, cost are captured and displayed in real-time side-by-side comparison table
  7. Dashboard output is stored in analysis_history with context_snapshot={"model_id":"...","benchmark":true,"benchmark_meta":{...}}

Phase 2: Evaluate (Day 5+)

python -m src.services.model_benchmark --evaluate --days 5
  1. Calls BacktestService.run_backtest() which finds all pending analyses
  2. For each, fetches forward daily bars via DataFetcherManager
  3. BacktestEngine.evaluate_single() classifies each prediction as win/loss/neutral
  4. Results stored in backtest_results table

Phase 3: Report

python -m src.services.model_benchmark --report
  1. JOINs backtest_results ↔ analysis_history to get model_id for each result
  2. Groups results by model_id, computes per-model metrics
  3. Calculates composite score, ranks models
  4. Outputs text leaderboard and JSON report, saves to reports/

All-in-One

python -m src.services.model_benchmark --full --stocks AAPL,TSLA

Runs all three phases sequentially. Useful for immediate testing when eval window is 0 (same-day analysis + evaluation).

Phase 0: Passive Benchmarking (Zero-Click Background Collection)

核心设计理念：零操作、全自动

用户无需显式调用 benchmark CLI。每次通过 main.py 正常分析股票时，系统自动在 context_snapshot 中标注当前使用的模型 ID 和性能数据（延迟 / Token / 成本估算），回测评估后自动产出跨模型对比报告。

工作原理：

  1. 自动标注（BenchmarkTagger）：在 StockAnalysisPipeline 的每次分析执行前后，自动捕获 LLM 调用耗时（time.perf_counter()）、Token 用量（从 AgentResult.stats 提取）、模型 ID（从 Config.litellm_model 或 AgentResult.model 读取），写入 context_snapshot.benchmark_meta
  2. 被动积累：用户在日常使用中切换不同模型（今天用 Claude、明天用 Gemini），每条分析记录自动标注对应模型。日积月累，自然形成多模型对比数据集
  3. 自动报告：回测完成后，若 BENCHMARK_AUTO_REPORT=true，自动调用 ModelBenchmarkService.generate_report()，JOIN 查询所有带 "benchmark": true 标记的回测结果，产出模型排名报告并保存到 reports/benchmark_*.txt

配置项（.env）：

# 是否自动为每次分析标注模型+性能数据（默认 true）
BENCHMARK_AUTO_TAG=true
# 是否在回测完成后自动生成跨模型对比报告（默认 true）
BENCHMARK_AUTO_REPORT=true

Agent 与非 Agent 路径均覆盖：

  • 非 Agent 路径（analyzer.analyze()）：标注 LLM 耗时；Token 数据通常不可用（传统路径不暴露 usage），benchmark_meta.total_tokens=0
  • Agent 路径（executor.run()）：标注 LLM 耗时 + 完整 Token 数据 + 成本估算，性能对比数据最完整

与显式 benchmark CLI 的关系：

  • 显式 benchmark CLI（python -m src.services.model_benchmark --analyze ...）仍然保留，用于主动对比多个模型（在单次运行中同时调用多个模型分析同一只股票）
  • 被动标注（BenchmarkTagger）是互补机制，在日常使用中零成本收集数据，无需额外操作
  • 两种路径产出的 context_snapshot 格式完全一致，共用同一套 generate_report() 查询逻辑

5. Data Model & Storage

No new tables — reuse of existing schema

TableFieldUsage

analysis_history operation_advice The model's trading advice — "买入", "Buy", "观望", etc.
analysis_history stop_loss, take_profit Sniper points from dashboard → used by BacktestEngine for simulated exit

Report Query (conceptual)

SELECT bt.*, ah.context_snapshot
FROM backtest_results bt
JOIN analysis_history ah ON bt.analysis_history_id = ah.id
WHERE bt.eval_window_days = 5
  AND bt.eval_status = 'completed'
  AND ah.context_snapshot LIKE '%"benchmark": true%'

6. Model Override Mechanism

The system normally reads the primary model from LITELLM_MODEL env var at startup. To run the same analysis through multiple models without restarting the process, _ModelOverride is a context manager that:

  1. Saves current LITELLM_MODEL and AGENT_LITELLM_MODEL
  2. Sets both to the target model_id
  3. Calls setup_env(override=True) to reload Config with the new model
  4. On __exit__, restores original values and reloads again
Thread Safety

This mutates process-level environment variables. The parallel execution strategy (--parallel) mitigates this by grouping work items by model and processing model groups sequentially — only stocks within the same model group run concurrently via ThreadPoolExecutor. A threading.Lock guards env var mutations. Subprocess isolation is recommended for fully independent multi-model parallelism.

8. Edge Cases & Error Handling

No models configured

discover_models() returns empty list. CLI prints instructions for configuring LLM_CHANNELS or API keys.

Stock limit exceeded

Maximum 20 stocks per benchmark run. Prevents runaway API costs. Raises ValueError.

Insufficient forward data

BacktestEngine returns eval_status="insufficient_data". These records are excluded from model scoring but counted separately.

Model API failure during analysis

Per-stock exception caught; error logged to results["errors"]. Other (stock, model) combinations continue unaffected.

No benchmark-tagged results in database

generate_report() returns empty report with warning log. CLI prints "No benchmark data available."

Config reload side effects

_ModelOverride calls setup_env(override=True) which reloads dotenv. This may affect other loaded modules. Mitigation: benchmark CLI always runs as a fresh process invocation. Parallel mode groups work by model to minimize env-var thrash.

Rate limit throttling (--parallel mode)

Running multiple model invocations concurrently may trigger API rate limits (especially for OpenAI and Anthropic). Mitigations: (a) use --max-parallel to limit concurrency (default 3), (b) models are still processed sequentially — only stocks within one model are parallelized, (c) fall back to sequential mode if rate-limit errors persist.

Cost estimation accuracy

Cost estimates use a hardcoded pricing table (_MODEL_PRICING) and token counts from the agent result metadata. Actual API billing may differ due to cached tokens, batch pricing, or provider-specific discounts. Estimates should be treated as comparative (Model A vs Model B) rather than absolute dollar amounts.

9. Limitations & Future Work

Current Limitations

  • Process-level env mutation — _ModelOverride is not thread-safe. Parallel mode (--parallel) mitigates by grouping work items by model and processing model groups sequentially with a threading.Lock. Full multi-model parallelism would require subprocess isolation.
  • No cost tracking ✅ Implemented — Cost estimation via _MODEL_PRICING table, token capture from agent stats, cost-efficiency ranking in performance leaderboard. Limitations: hardcoded prices, no real-time billing API integration.
  • Sequential-only execution ✅ Implemented — --parallel flag now supports concurrent stock analysis via ThreadPoolExecutor with per-model grouping.
  • Fixed eval window — All stocks share the same 5-day window. In practice, volatile vs stable stocks may need different horizons.
  • No statistical significance test — Current ranking is purely score-based. Small sample sizes may produce misleading rankings.
  • Single run per (stock, model) — LLM outputs have inherent randomness. Multiple runs per combination would give confidence intervals.
